## My common mistake patterns

1. **Missing articles** (the/a/an) — Korean has none, so I skip them
2. **"every" + plural** — "every functions" should be "every function" (always singular)
3. **Vague verbs** — using "do" instead of specific verbs like manage, handle, implement
4. **Korean word order** — topic-object-verb bleeds into English

#### 2026-03-16
Said: "There is always chance to add new md files"
Fix: "There's always a chance that new MD files get added"
Why: Need article "a" before "chance." Use passive "get added" for system events.

#### 2026-03-16
Said: "I think there is need some comprehensive order"
Fix: "I think there needs to be a comprehensive order"
Why: "There is need" is awkward. Standard construction is "there needs to be."

#### 2026-03-16
Said: "to read all the md files in docs directory"
Fix: "for reading all the MD files in the docs directory"
Why: "For + gerund" after "order." Add "the" before specific directory name.

#### 2026-03-16
Said: "How can we develop one product with multiple sessions"
Fix: "How do you develop one product with multiple sessions?"
Why: "How can we" implies doubt. "How do you" asks about standard practice — more confident.

#### 2026-03-16
Said: "How the sexiest big tech engineers do 100 person dev"
Fix: "How top-tier big tech engineers handle 100-person development"
Why: "Sexiest" doesn't apply to engineers professionally. Use "top-tier" or "elite." Replace vague "do" with specific "handle."

#### 2026-03-16
Said: "I don't know how to architect and build"
Fix: "I'm not confident in architecting and building this from scratch"
Why: "I don't know how" = zero skill. "Not confident in" = can learn. Much better positioning.

#### 2026-03-16
Said: "every single functions should be made as MD file"
Fix: "Every function should be defined in an MD file"
Why: No "s" after "every" — always singular. "Made as" → "defined in" is more precise.

#### 2026-03-16
Said: "so can I build web/app based tmux system"
Fix: "Is it possible to build a web-based tmux alternative?"
Why: "Can I" is informal. "Is it possible to" is professional for feasibility questions.

#### 2026-03-16
Said: "vibe coders should test every functions right away without setting some tools"
Fix: "Vibe coders should be able to test every feature immediately without configuring any tools"
Why: "Functions" → "feature" (user-facing). "Right away" → "immediately." "Setting some" → "configuring any."

#### 2026-03-16
Said: "I will not think of me as developer"
Fix: "I shouldn't think of myself as a developer"
Why: Reflexive pronoun "myself" needed. "Shouldn't" is softer than "will not."

#### 2026-03-16
Said: "I also want to memorize every trouble shooting"
Fix: "I also want to document every troubleshooting pattern"
Why: "Memorize" → "document" (building reference, not rote learning). "Troubleshooting" is one word. Add "pattern."

#### 2026-03-16
Said: "claude.md에도 추가해야하지않나 LEARNING에 대해서"
Fix: "Shouldn't we also add a reference to LEARNING.md in CLAUDE.md?"
Why: "Shouldn't we also" is collaborative proposal style — standard in Toronto dev meetings.

#### 2026-03-16
Said: "But for my learning... I want to learn Go too"
Fix: "For my own growth, I want to learn Go as well"
Why: "For my learning" → "for my own growth" sounds more intentional. "Too" → "as well" at end is more polished.

#### 2026-03-16
Said: "What is the best length of CLAUDE or other md files"
Fix: "What's the ideal length for CLAUDE.md and the other spec files?"
Why: "Best length" → "ideal length" is more professional. Add "spec files" for specificity.

#### 2026-03-16
Said: "the learning histories always loged with best length"
Fix: "The learning entries should always be logged at an optimal length"
Why: "Histories" → "entries" (specific unit). "Loged" → "logged" (double g). "Best" → "optimal."

## Professional phrases to practice

### Standup
"Yesterday I shipped the session management API. Today I'm wiring up the WebSocket integration. No blockers."

### Code review
"Left a few comments — nothing blocking, mostly style nits. LGTM overall."

### Architecture discussion
"Let's not over-engineer this. The simplest approach that works is the right one at this stage."

### Asking for help
"I've been debugging this for an hour. Here's what I've tried so far. Any ideas what I might be missing?"
