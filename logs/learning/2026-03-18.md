#### Web Terminal Architecture (xterm.js + PTY + tmux)
```
Browser (xterm.js) ↔ WebSocket ↔ FastAPI ↔ PTY (pty.openpty) ↔ tmux ↔ bash
```
- **xterm.js**: open-source terminal emulator for browsers (industry standard)
- **WebSocket**: binary frames for keystrokes, text frames for resize/ping commands
- **PTY (pseudo-terminal)**: pty.openpty() creates master/slave pair — gives real terminal behavior (line discipline, job control, colors)
- **tmux**: session persistence — survives page refresh/disconnect
- Same architecture as Gitpod, GitHub Codespaces, VS Code Web
- We built the WebSocket↔PTY↔tmux glue, not xterm.js itself

#### PTY implementation failures and lessons
1. **os.fork()**: crashes inside uvicorn async event loop (I/O error)
2. **asyncio subprocess PIPE**: no real PTY → no line discipline, `\n` not converted to `\r\n`, no job control, garbled output
3. **pty.openpty() + subprocess.Popen**: CORRECT approach — real PTY pair, Popen uses slave fd, parent reads master fd via run_in_executor
- Key: `preexec_fn=os.setsid` for proper process group
- Key: `fcntl.F_SETFL, flags | os.O_NONBLOCK` for async reads
- Key: `TIOCSWINSZ` ioctl for terminal resize

#### How Docker volume mounts work
```
docker-compose.yml:
  volumes:
    - .:/app          ← host source code mounted INTO container
    - workspace_data:/workspace   ← persistent Docker volume
```
- `.:/app` means host directory `c:\Sources\Factory001_DDalKKak\` IS `/app` inside container
- When you edit code on host → container sees changes instantly (no rebuild needed)
- uvicorn `--reload` watches for file changes → auto-restarts server

- `docker-compose restart api` needed when auto-reload doesn't trigger

#### How WebSocket connections work
```
HTTP: request → response → connection closed (one-shot)
WebSocket: connection stays OPEN → both sides send messages anytime
```
- Browser opens WebSocket: `new WebSocket("ws://localhost:8000/ws/terminal/abc123")`
- Server accepts: `await websocket.accept()`
- Browser sends keystrokes: `ws.send(bytes)` → server receives → writes to PTY
- Server sends output: `await websocket.send_bytes(data)` → browser receives → xterm renders
- Connection stays open until page closes or network drops
- Used for: terminal I/O, real-time session progress updates

#### How xterm.js copy/paste works in browser
- `term.onSelectionChange()` → fires when user drags to select text
- `navigator.clipboard.writeText(sel)` → copies to system clipboard (browser API)
- `term.attachCustomKeyEventHandler()` → intercepts Ctrl+C, Ctrl+V before they go to PTY
- Ctrl+C with text selected → copy (not SIGINT)
- Ctrl+C without selection → sends \x03 (SIGINT) to PTY → kills running process
- Ctrl+V → reads from clipboard → sends text to PTY stdin
- Browser limitation: Cmd+T, Cmd+W captured by browser, can't override

#### Session lifecycle in DalkkakAI
```
created → queued → running → review → merged
                    ↓         ↓
                   error    conflict
                    ↓
                  (retry → queued again)
```
- **created**: DB record exists, git worktree created (Auto AI only)
- **queued**: waiting for queue worker to pick up (10s poll interval)
- **running**: executor is calling Claude API in a loop
- **review**: agent finished, tests passed, waiting for user approval
- **merged**: branch merged into main, worktree cleaned up
- **error**: something failed (API error, test failure, etc.)
- Terminal sessions skip: created → running (no queue, no executor)

#### How the session queue worker works
```python
# backend/sessions/queue.py — runs as asyncio background task
async def run_queue_worker():
    while True:
        await asyncio.sleep(10)  # poll every 10 seconds
        # For each user with queued sessions:
        #   1. Check concurrency limit (free=1, starter=2, growth=5, scale=10)
        #   2. If under limit, pick highest-priority queued session
        #   3. Dispatch to AgentExecutor (Claude API tool-use loop)
        #   4. Update session status: running → review or error
```
- Runs inside FastAPI lifespan (started on app startup, cancelled on shutdown)
- Concurrency-aware: won't start more sessions than the user's plan allows
- Terminal sessions are filtered out (agent_type != "terminal")

#### What tmux actually is
- **tmux** = "terminal multiplexer" — runs terminal sessions that persist independently
- Without tmux: close terminal → process dies
- With tmux: close terminal → process keeps running in background
- `tmux new-session -d -s name` → create detached session
- `tmux attach-session -t name` → connect to existing session
- `tmux has-session -t name` → check if session exists
- Our use: WebSocket disconnect = detach. Reconnect = re-attach. Process never dies.
- Originally built for SSH — disconnect from server, process keeps running

#### Docker bind mount vs named volume
```
Named volume:  workspace_data:/workspace
  → Docker manages storage internally
  → Files NOT visible on host filesystem
  → Good for production (isolated)

Bind mount:    ./workspace:/workspace
  → Host directory mounted INTO container
  → Files visible at C:\Sources\...\workspace\
  → Good for development (can inspect AI-generated code)
```
- We switched to bind mount so the Files tab can show real code
- `/workspace/{startup_id}/` contains the actual startup code
- Added `/workspace/` to .gitignore — AI-generated code shouldn't be committed to our repo

#### Path traversal security
- When exposing file read via API, MUST prevent `../../etc/passwd` attacks
- Solution: `os.path.realpath()` resolves symlinks and `..` to absolute path
- Check: `if not real_path.startswith(workspace_root)` → block
- This is OWASP top 10 — always validate file paths at system boundaries
